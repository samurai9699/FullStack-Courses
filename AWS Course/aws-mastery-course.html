What is AWS?

Amazon Web Services is the world's largest cloud platform. This module demystifies the cloud and explains why AWS dominates the industry.

~20 min read
Beginner
1 Quiz

☁️ The Cloud, Simply Explained

Before AWS existed, companies had to buy physical servers, put them in a room, hire people to manage them, and hope nothing broke. This was expensive, slow, and painful.

The cloud is simply someone else's computers — specifically Amazon's — that you rent over the internet. Instead of buying a server for $10,000, you pay $0.02/hour for one. You use it for 3 hours and pay $0.06. Done.

Real analogy: You don't own a power plant to get electricity. You just plug in and pay for what you use. AWS is the power plant for computing.
⚡
On-demand
Spin up a server in 60 seconds. No waiting weeks for hardware delivery.
📏
Scalable
Handle 10 users or 10 million. Scale up or down instantly.
💰
Pay-as-you-go
Only pay for what you actually use. No upfront capital costs.
🌍
Global
Deploy apps close to your users anywhere in the world.

📊 Why AWS and Not Others?

AWS launched in 2006 — years before Google Cloud or Azure — and now holds the largest cloud market share. Most companies use it, meaning most job listings mention it.

ProviderMarket ShareBest Known For
AWS~31%Broadest service catalog, enterprise adoption
Azure~25%Microsoft ecosystem, Active Directory
Google Cloud~11%Data/ML services, Kubernetes origin
For your first job: AWS skills are the most transferable. Learning AWS concepts teaches you cloud fundamentals that apply everywhere.

🗺️ The AWS Console

The AWS Console is the web UI at console.aws.amazon.com. Think of it as a control panel for all your cloud resources. When you start a job, you'll use this dashboard daily.

1
Create a Free Account
Go to aws.amazon.com. The Free Tier gives you 750 hours/month of EC2, 5GB of S3, and an RDS instance — enough to learn everything in this course without paying.
2
Understand the Dashboard
The top search bar lets you find any service. The region selector (top right) shows where your resources live. The services menu groups everything by category: Compute, Storage, Database, etc.
3
Bookmark These Services
EC2, S3, RDS, IAM, VPC. These five cover 90% of what you'll encounter in a junior/mid-level role.

AWS Core Concepts

Regions, Availability Zones, VPCs — the building blocks everything else runs on. Understanding this is the difference between a junior and a mid-level engineer.

~30 min read
Foundational
1 Quiz

🌍 Regions & Availability Zones

AWS has physical data centers all over the world. They're organized into a two-level hierarchy you must understand.

┌─────────────────────────────────────────────────┐ │ AWS REGION (e.g. us-east-1) │ │ │ │ ┌──────────────┐ ┌──────────────┐ │ │ │ AZ: us- │ │ AZ: us- │ ... │ │ │ east-1a │ │ east-1b │ │ │ │ │ │ │ │ │ │ [DataCenter]│ │ [DataCenter]│ │ │ │ [DataCenter]│ │ [DataCenter]│ │ │ └──────────────┘ └──────────────┘ │ │ Connected via high-speed private links │ └─────────────────────────────────────────────────┘
🌍 What is a Region? ▼
A Region is a geographic location that contains multiple data centers. Examples: us-east-1 (N. Virginia), eu-west-1 (Ireland), ap-southeast-1 (Singapore). When you create a resource like an EC2 instance, it lives in a specific Region. Choose the Region closest to your users for the lowest latency.
🏢 What is an Availability Zone (AZ)? ▼
An AZ is one or more physical data centers within a Region. Each Region has 2–6 AZs. They're physically separated (different buildings, different power supplies, different flood zones) but connected with very fast, low-latency networking. The magic: if one AZ goes down due to a fire or power failure, your app keeps running in another AZ. This is called high availability.
🔣 Region naming convention ▼
Regions follow the pattern: [area]-[direction]-[number].

• us-east-1 → United States, East coast, first region
• eu-west-2 → Europe, West, second region (London)
• ap-southeast-1 → Asia Pacific, Southeast (Singapore)

AZs append a letter: us-east-1a, us-east-1b, us-east-1c
Interview gotcha: "AZ" and "Region" are NOT interchangeable. An AZ is inside a Region. A Region has multiple AZs. Confusing these in an interview is an instant red flag.

🔒 VPC — Your Private Network

A Virtual Private Cloud (VPC) is an isolated virtual network you control. Think of it as your own private section of AWS where you decide who can access what.

Every AWS account gets a default VPC automatically. Inside a VPC, you create subnets — smaller network chunks that live inside an AZ.

🌐
Public Subnet
Has a route to the internet. Your web servers and load balancers go here. Users can reach them.
🔒
Private Subnet
No direct internet access. Your app servers and databases go here. Only your infrastructure can reach them.
🚪
Internet Gateway
The "door" that connects your VPC to the public internet. Attach one to enable internet access.
🛣️
Route Table
Rules that tell network traffic where to go. Each subnet has one.

🔀 NAT Gateway & VPC Endpoints

Private subnets can't reach the internet by default. But sometimes they need to — to download packages, pull container images, or call external APIs. That's what NAT Gateways solve.

🔄
NAT Gateway
Lives in a public subnet. Lets private subnet resources reach the internet (outbound only) while blocking inbound connections. Essential for ECS tasks pulling images from ECR.
🔌
VPC Gateway Endpoint
Free. Routes traffic to S3 and DynamoDB through the AWS network backbone — never touching the public internet. Faster and cheaper.
🔗
VPC Interface Endpoint
Creates an elastic network interface (ENI) in your subnet for any AWS service (SQS, SNS, ECR, CloudWatch, etc.). Keeps traffic private. Costs per hour + per GB.
🛡️
NACL vs Security Group
NACLs are stateless firewalls at the subnet level. Security Groups are stateful firewalls at the instance level. Always use Security Groups for application-level control.
Cost tip: NAT Gateways cost ~$0.045/hour + data processing. For dev environments, consider NAT instances (cheaper) or VPC endpoints to reduce NAT traffic.

🏗️ The Global vs Regional vs AZ Pattern

Different AWS services operate at different scopes. This trips up beginners constantly.

ScopeServicesWhat it means
GlobalIAM, Route 53, CloudFrontNot tied to any region — applies everywhere
RegionalS3, VPC, RDS, SQS, ECSCreated in a specific region, replicated within it
AZ-specificEC2 instances, EBS volumes, ElastiCache nodesLive inside one AZ — if AZ fails, so does this resource
Why this matters: S3 is regional and automatically replicates across AZs. But your single EC2 instance is in one AZ. If that AZ has issues, your EC2 is down. That's why production apps run EC2 in multiple AZs.

IAM & Security

Identity and Access Management — who can do what in your AWS account. Getting IAM wrong is how breaches happen. Getting it right is how you impress in interviews.

~25 min read
Critical
1 Quiz

🔑 IAM Concepts

IAM is global (not per region) and controls everything. There are four key building blocks:

👤
Users
A person or application that needs AWS access. Has credentials (password or access keys).
👥
Groups
A collection of users. Attach a policy to a group and all users in it get those permissions.
📋
Policies
JSON documents that define what is ALLOWED or DENIED. Attached to users, groups, or roles.
🎭
Roles
Temporary identity for AWS services or external users. An EC2 instance assumes a Role to access S3 — no hard-coded passwords needed.

📝 IAM Policies Explained

A policy is just a JSON document. Here's a real one that allows listing and reading an S3 bucket:

JSON — IAM Policy
{
  "Version": "2012-10-17",
  "Statement": [
    {
      "Effect": "Allow",
      "Action": [
        "s3:GetObject",
        "s3:ListBucket"
      ],
      "Resource": [
        "arn:aws:s3:::my-bucket",
        "arn:aws:s3:::my-bucket/*"
      ]
    }
  ]
}
Default Deny: IAM denies EVERYTHING by default. You must explicitly Allow each action. If you see a permission error in AWS, your IAM policy is missing an Allow for that action.

🎭 IAM Roles — The Right Way

One of the most important concepts in AWS security: never hard-code AWS credentials in your code. Instead, use IAM Roles.

1
Create an IAM Role with permissions
e.g., a Role called MyAppRole with an S3 read policy attached.
2
Assign the Role to your EC2 instance
When you launch EC2, you select the IAM Role. The instance "becomes" that Role.
3
Your app uses the Role automatically
The AWS SDK automatically fetches temporary credentials from the instance metadata. Zero passwords in your code.
Never do this: Storing AWS_ACCESS_KEY_ID and AWS_SECRET_ACCESS_KEY in your source code or .env files that get committed to GitHub. This is how thousands of accounts get hacked each year. Use Roles for services, environment variables only for local dev.

🛡️ Principle of Least Privilege

This is the most important IAM concept to mention in any interview:

Give only the permissions actually needed — nothing more. If your app only reads from S3, don't give it s3:* (all S3 permissions). Give it only s3:GetObject.

Interview line: "I follow the principle of least privilege — every IAM Role or User gets only the permissions needed for their specific task, scoped to specific resources using ARNs."

EC2 — Elastic Compute

EC2 is AWS's virtual server service. You'll use it to host applications, run background jobs, and build anything that needs dedicated compute power.

~35 min read
Core Service
Hands-on + Quiz

💻 What is an EC2 Instance?

An EC2 instance is a virtual machine (VM) running in AWS data centers. You choose the operating system, CPU, RAM, and storage. You SSH into it like any Linux server and run your app.

Elastic in the name means you can scale up or down. Start with 1 small instance, and spin up 50 more in minutes if traffic spikes.

📦 Instance Types — Choosing the Right Size

AWS has hundreds of instance types. The naming convention tells you everything:

t3.micro │ │ │ │ │ └── Size: nano, micro, small, medium, large, xlarge, 2xlarge... │ └──── Generation: 1, 2, 3, 4... (higher = newer, better perf/price) └────── Family: t (burstable), m (general), c (compute), r (memory), g (GPU)
FamilyOptimized ForUse Case
t3/t4gBurstable (cheap)Dev/test, small web apps, Free Tier
m6i/m7iGeneral purposeMost production web apps and APIs
c6i/c7iCPU-heavyVideo processing, batch jobs, high-traffic APIs
r6i/r7iMemory-heavyIn-memory databases (Redis), big data analytics
g4dn/g5GPUML training/inference, video rendering
For the Free Tier: Use t2.micro or t3.micro. 750 hours/month free for 12 months.

🔥 Security Groups — EC2's Firewall

A Security Group is a virtual firewall that controls inbound and outbound traffic for your EC2 instance. It works at the instance level.

Rule TypeProtocolPortSource/DestPurpose
InboundTCP22Your IPSSH access — NEVER 0.0.0.0/0
InboundTCP800.0.0.0/0HTTP (public web traffic)
InboundTCP4430.0.0.0/0HTTPS (public web traffic)
OutboundAllAll0.0.0.0/0Allow all outbound (default)
Critical security rule: Never open SSH (port 22) to 0.0.0.0/0. Bots will find open SSH ports and attempt brute-force attacks within minutes.

🚀 Launching Your First EC2 Instance

1
Open EC2 Console → Launch Instance
Search for EC2 in the console, click "Launch instance".
2
Choose AMI
Select "Amazon Linux 2023 AMI" (marked as Free Tier eligible).
3

4
Create a Key Pair
Choose RSA, .pem format. Download and save it — you cannot download it again. Store at ~/.ssh/.
5
Configure Security Group

6

ALB & Load Balancing

Application Load Balancers distribute incoming traffic across multiple targets — EC2 instances, containers, or Lambda functions. They're the front door of every production architecture.

~25 min read
Core Service
1 Quiz

⚖️ Why You Need a Load Balancer

Without a load balancer, users connect directly to a single EC2 instance. If that instance dies, your app is down. If traffic spikes, that single instance gets overwhelmed.

A load balancer sits in front of your instances and distributes traffic. If one instance is unhealthy, the ALB stops sending traffic to it. You can add or remove instances without users noticing.

Users │ ▼ ┌─────────────────────────┐ │ Application Load │ │ Balancer (ALB) │ │ (public subnet) │ └──────┬──────────┬───────┘ │ │ ┌────▼───┐ ┌───▼────┐ │ EC2-a │ │ EC2-b │ ← Target Group │(AZ-a) │ │(AZ-b) │ └────────┘ └────────┘

🔀 ALB vs NLB vs CLB

TypeLayerBest ForKey Feature
ALBLayer 7 (HTTP/HTTPS)Web apps, APIs, microservicesPath-based & host-based routing, WebSocket support
NLBLayer 4 (TCP/UDP)Ultra-low latency, gaming, IoT, non-HTTP protocolsStatic IP, millions of requests/sec
CLBLayer 4 + 7Legacy — do not use for new projectsDeprecated, limited features
Rule of thumb: If your traffic is HTTP/HTTPS, use an ALB. If you need raw TCP/UDP performance or static IPs, use an NLB. Never choose CLB for new architectures.

📋 Listeners, Rules & Target Groups

An ALB has three key concepts that work together:

👂
Listener
Checks for connection requests on a specific port (80 for HTTP, 443 for HTTPS). You can have multiple listeners per ALB.
📐
Rules
Conditions on a listener that route traffic differently. E.g., /api/* goes to backend targets, /images/* goes to S3.
🎯
Target Group
A group of registered targets (EC2 instances, IPs, Lambda functions). The ALB sends traffic to healthy targets in the group.
❤️
Health Check
The ALB periodically pings each target (e.g., GET /health). If a target fails 2 consecutive checks, it's marked unhealthy and removed from rotation.

🛣️ Path-Based Routing

This is the killer feature of ALB. You can route traffic to different target groups based on the URL path or hostname.

User Request: https://myapp.com/api/users │ ┌────▼────────────────────────────────┐ │ ALB Listener (443) │ │ │ │ Rule 1: /api/* → Backend TG │ │ Rule 2: /web/* → Frontend TG │ │ Rule 3: /auth/* → Auth Service TG │ │ Default: → Frontend TG │ └─────────────────────────────────────┘
Interview point: Path-based routing lets you build microservices behind a single ALB. Each service gets its own target group, and the ALB routes by URL prefix. This is the standard pattern for service-oriented architectures on AWS.

🔒 ALB + HTTPS with ACM

Never serve production traffic over plain HTTP. ALB integrates directly with AWS Certificate Manager (ACM) for free SSL/TLS certificates that auto-renew.

1
Request a certificate in ACM
Enter your domain name (e.g., myapp.com). Choose DNS validation.
2
Add a 443 listener to your ALB
Select the ACM certificate. Set the default action to forward to your target group.
3
Redirect HTTP → HTTPS
On the port 80 listener, set action to "Redirect to HTTPS". All HTTP requests automatically get a 301 redirect.

Auto Scaling Groups

ASGs automatically add or remove EC2 instances based on demand. Traffic spike at 3 AM? New instances spin up. Traffic dies down? Instances terminate. You pay only for what you need.

~25 min read
Core Service
1 Quiz

📈 How Auto Scaling Works

An ASG manages a fleet of EC2 instances. You define the minimum, desired, and maximum number of instances. AWS maintains the desired count automatically.

Traffic spike detected │ ▼ CloudWatch Alarm: CPU > 70% │ ▼ ASG: scale from 2 → 4 instances │ ▼ 2 new EC2 instances launch (60s boot) │ ▼ ALB health check passes → instances receive traffic │ ▼ CPU drops below 30% → ASG scales back to 2
⬇️
Min Size
The minimum number of instances running at all times. Never goes below this, even during low traffic.
🎯
Desired Capacity
The target number of instances. The ASG always tries to maintain this. Scaling policies change the desired capacity.
⬆️
Max Size
The ceiling. The ASG will never launch more than this number, protecting you from runaway costs.
🏥
Health Check Grace Period
How long to wait after an instance launches before checking its health. Default: 300 seconds. Gives the app time to start.

📐 Scaling Policies

PolicyHow It WorksBest For
Target TrackingSet a target metric (e.g., CPU = 50%). ASG adjusts instances to maintain it.Most workloads. Simple and effective. Start here.
Step ScalingDefine thresholds: CPU > 60% add 1, CPU > 80% add 3.Complex workloads with non-linear scaling needs.
Scheduled ScalingScale at specific times. E.g., add instances Mon-Fri 8 AM.Predictable traffic patterns like business hours.
Predictive ScalingML-based. AWS forecasts traffic and pre-scales.Cyclical workloads with historical data.
Recommended starting config: Target tracking on CPU utilization at 50%, with a min of 2 and max of 10. This handles 90% of web application scenarios.

🔧 Launch Templates

A Launch Template defines the "recipe" for each instance the ASG creates: AMI, instance type, security group, IAM role, user data script, storage, etc.

JSON — Launch Template (conceptual)
{
  "ImageId": "ami-0abcdef1234567890",
  "InstanceType": "t3.small",
  "KeyName": "my-key",
  "SecurityGroupIds": ["sg-0123456789"],
  "IamInstanceProfile": { "Name": "MyAppRole" },
  "UserData": "<base64 encoded bootstrap script>",
  "BlockDeviceMappings": [{
    "DeviceName": "/dev/xvda",
    "Ebs": { "VolumeSize": 30, "VolumeType": "gp3" }
  }]
}
Launch Templates vs Launch Configurations: Launch Configurations are legacy and cannot be modified after creation. Always use Launch Templates — they support versioning, overrides, and newer instance features.

🏗️ ALB + ASG — The Standard Production Pattern

This is the most common production architecture on AWS. The ALB distributes traffic and reports health; the ASG maintains instance count.

Internet │ ▼ ┌──────────────┐ │ ALB │ ← Public subnet, HTTPS listener │ Health Check │ pings GET /health └──┬───────┬───┘ │ │ ┌──▼──┐ ┌─▼───┐ │EC2-1│ │EC2-2│ ← ASG manages these │(a) │ │(b) │ Launches in private subnets across AZs └─────┘ └─────┘ ▲ ▲ └───────┘ ASG: Min=2, Max=10 Policy: Target tracking CPU=50%

💰 Mixed Instances & Spot in ASG

Mixed Instances Policies let you blend On-Demand and Spot instances in a single ASG, cutting costs dramatically while maintaining availability.

Cost optimization pattern: Set base capacity of 2 On-Demand instances, then allow up to 8 additional Spot instances from diverse instance types (t3, m5, m6i). If one Spot type gets interrupted, the ASG replaces it with another. This can cut EC2 costs by 50-70%.

S3 — Object Storage

Simple Storage Service. Unlimited storage for files, images, videos, backups, and static websites. One of AWS's oldest and most reliable services.

~30 min read
Core Service
Code + Quiz

🗂️ Buckets & Objects

S3 has two main concepts: buckets (containers) and objects (files). Think of a bucket as a folder and objects as the files inside.

S3 is NOT a filesystem. It's object storage. There are no true directories — "folders" in S3 are just a naming convention using slashes in the key name, e.g. photos/2024/jan/img1.jpg.
🪣
Bucket
Top-level container. Name must be globally unique across ALL of AWS. Lives in one Region.
📄
Object
A file stored in a bucket. Has a key (path), value (data), metadata, and optional version ID.
🔑
Object Key
The "path" to the object: images/profile/user123.jpg. The key is unique within a bucket.
📏
Size Limits
Max object size: 5TB. Use multipart upload for anything over 5GB.

💾 Storage Classes

ClassAccess PatternRetrieval TimeCost
S3 StandardFrequent accessMillisecondsHighest storage cost
S3 Standard-IAInfrequent accessMillisecondsLower storage, retrieval fee
S3 Intelligent-TieringUnknown patternMillisecondsAuto-moves between tiers

🌐 Static Website Hosting

S3 can serve static HTML/CSS/JS files as a website. This is perfect for React/Vue/Next.js static exports, and costs pennies per month.

1
Create a bucket with your domain name
Name the bucket exactly your domain: www.myapp.com
2
Enable static website hosting
Bucket Properties → Static website hosting → Enable. Set index document: index.html
3
Disable Block Public Access & add bucket policy
Allow s3:GetObject to Principal: "*" for your bucket ARN.
4
Upload your build files
Upload your dist/ or build/ folder. Your site is now live at the S3 endpoint.
Production upgrade: Put CloudFront (AWS CDN) in front of your S3 bucket for HTTPS, custom domain support, and global caching.

⚙️ Working with S3 via CLI & SDK

Bash — AWS CLI S3 Commands
# Create a bucket
aws s3 mb s3://my-unique-bucket-name --region us-east-1

# Upload a file
aws s3 cp ./image.png s3://my-bucket/images/image.png

# Sync a local folder to S3 (great for deployments)
aws s3 sync ./dist s3://my-bucket/ --delete

# List bucket contents
aws s3 ls s3://my-bucket/

# Generate a pre-signed URL (valid 1 hour)
aws s3 presign s3://my-bucket/private-doc.pdf --expires-in 3600

🔄 Versioning & Lifecycle Rules

Versioning: Enable on a bucket to keep all versions of objects. Accidentally delete or overwrite a file? Restore the previous version.

Lifecycle Rules: Automatically move objects to cheaper storage classes or delete them after a set time. Example: move to Standard-IA after 30 days, archive to Glacier after 90 days, delete after 365 days.

RDS — Managed Databases

Relational Database Service. AWS handles patching, backups, replication. You just write SQL and connect.

~30 min read
Core Service
Code + Quiz

🗄️ What RDS Does For You

Running your own database on EC2 means you're responsible for everything: OS patches, database engine updates, backups, hardware failures, replication setup, monitoring. That's a second job.

RDS is a managed service — AWS handles all of that. You get a database endpoint and connect to it.

🔧
Automated Patching
AWS patches the OS and database engine automatically during your maintenance window.
📸
Automated Backups
Daily backups + transaction logs. Restore to any point in time within your retention period.
🔄
Multi-AZ Failover
Standby replica in another AZ. If primary fails, automatic failover in ~60 seconds.
📊
Read Replicas
Read-only copies to distribute SELECT queries. Scale reads independently from writes.

🗃️ Supported Database Engines

EngineBest ForNotes
PostgreSQLGeneral purpose, JSON, complex queriesMost popular choice for new SaaS apps
MySQLWeb apps, WordPress, legacy systemsHuge community, well-understood
Aurora (PostgreSQL)High-performance, scalableAWS-proprietary, 3x faster than standard PostgreSQL
Aurora (MySQL)High-performance, scalable5x faster than standard MySQL
SQL Server.NET apps, Microsoft ecosystemExpensive licensing
For new projects: Choose RDS PostgreSQL or Aurora PostgreSQL. PostgreSQL has excellent JSON support, full-text search, and extensions like PostGIS.

🏗️ Multi-AZ vs Read Replicas

MULTI-AZ (for availability/failover) READ REPLICA (for performance) ───────────────────────────────── ──────────────────────────────── App → Primary DB (AZ-a) ──sync──→ Standby App (writes) → Primary DB ↓ failover if primary fails ↓ replication Standby becomes new Primary App (reads) → Read Replica 1 App (reads) → Read Replica 2 • Standby is NOT readable • Replicas ARE readable • Automatic failover ~60s • You route read traffic manually • Same region, different AZ • Can be in different regions
Interview trap: "Can you read from the Multi-AZ standby?" Answer: NO. The standby is purely for failover. Only Read Replicas accept read traffic.

🔒 RDS Security — The Right Pattern

Internet │ ▼ [Load Balancer] ← Public Subnet │ ▼ [EC2 / App Server] ← Private Subnet │ ▼ [RDS Instance] ← Private Subnet (no public IP) │ Security Group: only accept connections from App Server SG on port 5432
1
Put RDS in a private subnet
Select private subnets via a DB Subnet Group. Disable "Publicly accessible".
2
Create an RDS Security Group
Inbound: allow port 5432 (Postgres) or 3306 (MySQL) ONLY from the app server's Security Group.
3
Use Secrets Manager for credentials
Never hard-code DB passwords. Store them in AWS Secrets Manager. RDS can rotate them automatically.
4
Enable encryption at rest
Enabled by default on new RDS instances. Uses AWS KMS. No performance penalty.

ElastiCache

In-memory caching that makes your database 10-100x faster for reads. ElastiCache sits between your app and RDS, serving frequently accessed data from memory instead of disk.

~20 min read

1 Quiz

⚡ Why Caching Matters

Every database query takes time — even optimized ones. If 10,000 users request the same product page, querying the database 10,000 times is wasteful. Caching stores the result in memory and serves it in under 1ms.

Without Cache: With Cache: App → RDS (5-20ms per query) App → ElastiCache (0.5ms) → hit? return │ ▼ miss RDS (5-20ms) → store in cache → return

🔄 Redis vs Memcached

FeatureRedisMemcached
Data StructuresStrings, Lists, Sets, Sorted Sets, HashesStrings only
PersistenceYes (snapshots + AOF)No — data lost on restart
ClusteringYes — multi-node with shardingYes — multi-threaded, sharding
Pub/SubYesNo
Sorted dataYes (sorted sets for leaderboards, queues)No
Best ForSession storage, leaderboards, queues, cachingSimple key-value caching, multi-threaded performance
Default choice: Pick Redis unless you have a specific reason for Memcached. Redis is more feature-rich, supports persistence, and has better tooling. Most AWS architectures use Redis.

📋 Caching Strategies

💤
Lazy Loading (Cache-Aside)
App checks cache first. On miss, query DB, store result in cache with TTL. Most common pattern. Simple and effective.
✏️
Write-Through
App writes to cache AND database simultaneously. Data is always fresh in cache, but writes are slower.
⏰
TTL (Time to Live)
Every cached entry has an expiration. After TTL, the entry is deleted and the next request hits the DB. Prevents stale data.
🗑️
Cache Invalidation
When data changes, delete the cached entry. Next read repopulates it. The hardest problem in caching is knowing when to invalidate.
JavaScript — Lazy Loading with Redis
import Redis from "ioredis";

const redis = new Redis(process.env.REDIS_URL);

async function getProduct(productId) {
  // 1. Check cache
  const cached = await redis.get(`product:${productId}`);
  if (cached) return JSON.parse(cached);

  // 2. Cache miss — query database
  const result = await query("SELECT * FROM products WHERE id = $1", [productId]);

  // 3. Store in cache with 5-minute TTL
  await redis.setex(`product:${productId}`, 300, JSON.stringify(result.rows[0]));

  return result.rows[0];
}

ElastiCache Redis nodes live in private subnets. Your app connects to the Redis primary endpoint for writes and the reader endpoint for reads (if using replicas).

┌──────────────────────────────────────────┐ │ App Servers (Private Subnet) │ │ │ │ │ ├──→ ElastiCache Redis (primary + │ │ │ replica in different AZ) │ │ │ │ │ │ │ cache miss │ │ │ ▼ │ │ └──→ RDS PostgreSQL (Multi-AZ) │ └──────────────────────────────────────────┘

ECS & Containers

Elastic Container Service. Run Docker containers on AWS without managing servers (Fargate) or on EC2 you control. ECS is the backbone of modern microservice architectures.

Core Service

🐳 What Are Containers?

A container packages your application code along with all its dependencies (libraries, runtime, system tools) into a single image. It runs identically on your laptop, in staging, and in production. No more "works on my machine."

Containers are lighter than VMs. An EC2 instance runs an entire OS. A container shares the host OS kernel and runs only your app. You can pack many containers onto one instance.

Key difference: A VM virtualizes hardware. A container virtualizes the operating system. VMs take minutes to start; containers start in seconds.

📦 ECS Core Concepts

🏢
Cluster
A logical grouping of compute resources (EC2 instances or Fargate). Your services and tasks run inside a cluster.
📋
Task Definition
The blueprint for your container: image, CPU, memory, ports, environment variables, IAM role, volumes. Think of it as a Docker Compose file for AWS.
🚀
Task
A running instance of a Task Definition. One container (or group of containers) actually executing. A Task is to a Task Definition what an EC2 instance is to an AMI.
🔄
Service
Maintains a desired number of running Tasks. If a task crashes, the Service replaces it. Integrates with ALB for load balancing. This is what you deploy to production.

🎭 Task Role vs Execution Role

This is a common confusion point. Two different IAM Roles are involved in ECS:

🚀
Task Execution Role
Used by the ECS agent to pull images from ECR, send logs to CloudWatch, and fetch secrets from Secrets Manager. You grant it: ecr:GetAuthorizationToken, ecr:BatchGetImage, logs:CreateLogStream, secretsmanager:GetSecretValue.
🔑
Task Role
Used by your application code running inside the container. If your app needs to read from S3, write to SQS, or call DynamoDB, those permissions go here. This is where least privilege matters most.
Common mistake: Granting S3 or DynamoDB permissions to the Execution Role instead of the Task Role. The Execution Role is for infrastructure tasks. The Task Role is for your application.

SQS & SNS

Messaging services that decouple your architecture. SQS queues messages for processing. SNS broadcasts notifications to multiple subscribers. Together they enable resilient, event-driven systems.

~25 min read
Architecture
1 Quiz

📨 Why Messaging?

Without messaging, Service A directly calls Service B. If Service B is slow or down, Service A fails too. Messaging breaks this dependency:

SYNCHRONOUS (tightly coupled): ASYNCHRONOUS (decoupled): Order Service → Payment Service Order Service → SQS Queue ← Payment Service • Payment is slow? Orders fail. • Payment is slow? Orders still accepted. • Payment is down? Orders fail. • Payment is down? Messages wait in queue. • Traffic spike? Both crash. • Traffic spike? Queue absorbs it.
Interview principle: "I decouple services with queues to improve resilience. If the downstream service is slow or unavailable, the queue buffers requests and the upstream service continues operating."

📬 SQS — Simple Queue Service

SQS is a fully managed message queue. Producers send messages to a queue. Consumers poll the queue and process messages.

FeatureStandard QueueFIFO Queue
OrderingBest-effort (may arrive out of order)Strictly ordered (First-In-First-Out)
DeliveryAt-least-once (duplicates possible)Exactly-once (no duplicates)
ThroughputUnlimited (nearly)300 msgs/sec (or 3,000 with batching)
Best forMost workloads. Order doesn't matter.Financial transactions, order processing where order matters.
⏱️ Visibility Timeout ▼
When a consumer receives a message, it becomes "invisible" to other consumers for the visibility timeout period (default 30 seconds). The consumer must delete the message after processing. If it doesn't (e.g., the consumer crashes), the message becomes visible again and another consumer picks it up. Set this to slightly longer than your expected processing time.
💀 Dead-Letter Queue (DLQ) ▼
A separate queue where messages go after failing to process a set number of times (maxReceiveCount). Without a DLQ, poison messages (messages that always cause errors) would be processed forever. With a DLQ, they're moved aside for investigation. Always configure a DLQ in production.
📨 Message Retention ▼
Messages are kept in the queue for 1-14 days (default: 4 days). If no consumer processes them within the retention period, they're deleted. For long-term storage, configure the DLQ with a longer retention period.

📢 SNS — Simple Notification Service

SNS is a pub/sub messaging service. You publish a message to a topic, and all subscribers receive it. Think of it as a broadcast system.

📡
Topic
A communication channel. You publish messages to a topic, and SNS delivers them to all subscribers.
📬
Subscribers
SQS queues, Lambda functions, HTTP endpoints, email addresses, SMS, or mobile push notifications.
🌊
Fan-Out Pattern
One SNS topic → multiple SQS queues. A single event (e.g., "order placed") triggers parallel processing: inventory update, payment processing, email notification.
🔒
Message Filtering
Subscribers can set filter policies to receive only specific messages. E.g., only receive messages where "type": "payment".

🏗️ SNS + SQS Fan-Out Architecture

S3 Event / API Gateway / App │ ▼ ┌──────────┐ │ SNS Topic│ ← "order-placed" event published here └──┬───┬───┘ │ │ ┌──▼┐ ┌▼────────┐ │SQS│ │SQS │ ← Each queue gets a copy │Pay│ │Inventory│ Processed independently └─┬─┘ └────┬────┘ │ │ ┌─▼─┐ ┌──▼──┐ │Lambda/│ │Lambda/│ │Worker │ │Worker │ └─────┘ └─────┘ Benefits: • Each service processes at its own pace • If payment processing fails, inventory still processes • Easy to add new subscribers (e.g., analytics) without changing existing services

⚙️ Sending Messages from Code

The exact questions you'll be asked, model answers, architecture design challenges, and key phrases that signal competence. This module covers all 14 prior topics.

❓ EC2 & ALB Questions

"What is the difference between stopping and terminating an EC2 instance?" ▼
Stop: Instance shuts down but EBS root volume is preserved. Not charged for compute, still pay for EBS storage. Can restart with data intact.

Terminate: Instance AND root EBS volume are deleted (unless Delete on Termination is false). Permanent.
"What's the difference between ALB and NLB?" ▼
ALB (Layer 7): Routes based on HTTP content — URL path, hostname, headers. Supports WebSocket, path-based routing, host-based routing. Best for web apps and APIs.

NLB (Layer 4): Routes TCP/UDP traffic. Ultra-low latency (~100ms vs ALB's ~400ms). Static IP addresses. Best for non-HTTP protocols, gaming, IoT, and extreme throughput.
"How would you ensure your EC2 app is highly available?" ▼
Deploy EC2 instances across multiple Availability Zones in an Auto Scaling Group. Put an ALB in front to distribute traffic and perform health checks. Use Launch Templates for consistent configuration. If one AZ fails, ASG maintains desired count in remaining AZs and ALB routes around the failure.

❓ Auto Scaling & Compute Questions

"What scaling policy would you use for a web application?" ▼
Target tracking on CPU at 50%. This is the simplest and most effective policy for most web apps. The ASG automatically adds instances when CPU exceeds 50% and removes them when it drops below. For more nuanced control, add scheduled scaling for known traffic patterns (e.g., business hours) and step scaling for traffic spikes that need faster response.
"What is the difference between Fargate and EC2 for ECS?" ▼
Fargate: Serverless — you don't manage instances. Pay per vCPU/memory per second. Start here for most workloads.

EC2: You manage the instances. Can use Spot for 70%+ savings. Better for GPU workloads or very high throughput where you need bin packing control. Choose EC2 when cost optimization with Spot is critical or you need GPU/ARM instances.

❓ S3 & Storage Questions

"How would you allow users to upload files directly to S3 from the browser?" ▼
Use pre-signed PUT URLs. Backend generates a pre-signed URL with a time limit (e.g., 10 minutes). Client uploads directly to S3 via PUT — no traffic through your server. After upload, S3 can trigger a Lambda via Event Notifications for post-processing (thumbnail generation, virus scan, etc.).
"What is S3's consistency model?" ▼
Since December 2020, S3 offers strong read-after-write consistency for all operations. Immediately after a successful PUT or DELETE, any subsequent GET or LIST reflects that change. No eventual consistency concerns.

❓ RDS & Caching Questions

"What is the difference between Multi-AZ and Read Replicas?" ▼
Multi-AZ = availability. Synchronous standby in another AZ. Automatic failover in ~60s. Standby is NOT readable. Same connection string.

Read Replicas = performance. Asynchronous replicas for read traffic. Up to 15 replicas. Can be cross-region. CAN be promoted to standalone if needed.
"When would you use ElastiCache?" ▼
When database reads are a bottleneck. Common scenarios: session storage (replace sticky sessions), query result caching (product pages, user profiles), rate limiting (count requests in Redis), leaderboards (sorted sets). Always use lazy loading (cache-aside) as the default pattern, with TTL to prevent stale data.

❓ Messaging & Event-Driven Questions

"When would you use SQS vs SNS?" ▼
SQS for point-to-point: one producer, one consumer. Messages are pulled by the consumer. Use when you need reliable processing with retry (DLQ) and want to decouple producer from consumer speed.

SNS for pub/sub: one producer, many subscribers. Messages are pushed to subscribers. Use when multiple services need to react to the same event. Often combined: SNS → fan-out → multiple SQS queues.
"What is a Dead-Letter Queue and why do you need one?" ▼
A DLQ is a separate SQS queue where messages go after failing to process a configured number of times (maxReceiveCount). Without a DLQ, "poison messages" that always cause errors would be retried forever, blocking the queue. With a DLQ, they're moved aside for investigation while the main queue continues processing healthy messages. Always configure one in production.

❓ Terraform & IaC Questions

"Why Terraform over CloudFormation?" ▼
Terraform is cloud-agnostic (works with AWS, GCP, Azure, 100+ providers). HCL is more readable than CloudFormation's verbose JSON/YAML. Terraform plan shows changes before applying. State management is explicit. Community modules are extensive.

CloudFormation is AWS-native, free, and deeply integrated (Drift Detection, StackSets). Better choice if you're 100% AWS and want zero external tooling. Most teams choose Terraform for its flexibility and ecosystem.
"How do you handle secrets in Terraform?" ▼
Never hard-code secrets in .tf files. Use AWS Secrets Manager or SSM Parameter Store to store secrets, and reference them in Terraform with data sources. For sensitive Terraform variables, use TF_VAR_ environment variables or .tfvars files excluded from Git via .gitignore. Enable state encryption and use a remote backend with restricted access.
