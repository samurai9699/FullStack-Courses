Why Docker?

You've heard the phrase "it works on my machine" — Docker kills that problem forever. It packages your app with everything it needs to run: the code, the runtime, the libraries, the config. Anywhere Docker runs, your app runs identically.

📦
Portability
Build once, run anywhere — dev laptop, CI pipeline, cloud server. Identical behavior.
⚡
Speed
Containers start in milliseconds. Spin up 50 services instantly. No VM overhead.
🔒
Isolation
Each container is sandboxed. Dependency conflicts between apps are gone.
📈
Scale
The foundation of microservices, CI/CD, and Kubernetes. Non-negotiable in 2025.

Containers vs. Virtual Machines

This is one of the most common interview questions. Know it cold.

FeatureVirtual MachineDocker Container
Boot timeMinutesMilliseconds
SizeGBs (full OS)MBs (shares host kernel)
IsolationFull OS isolationProcess-level isolation
PerformanceOverhead from hypervisorNear-native
PortabilityHeavy, disk imageLightweight layers
OSFull guest OS per VMShares host OS kernel
💡
Containers share the host OS kernel — they don't need to boot a whole operating system. This is why they're so fast and lightweight. VMs use a hypervisor to emulate hardware and run a full OS.

How Docker Is Structured

Docker has three main pillars you must understand:

1
Docker Engine (the daemon)
The background service that manages containers. It listens for API calls and handles the heavy lifting — building, running, distributing containers.
2
Docker CLI
The command-line tool you use to talk to the Docker daemon. When you run docker run, the CLI sends a request to the daemon over a REST API.
3
Docker Hub / Registry
The repository for Docker images. Like GitHub but for container images. You pull images from here to run, and push your own images to share.

Core Concepts

Before touching the CLI, get these terms locked in. Every Docker conversation relies on them.

Image

A Docker Image is a read-only template used to create containers. Think of it like a blueprint or a class in OOP. It contains your application code, dependencies, environment variables, and filesystem. Images are made of stacked, immutable layers.

✅
Analogy: An image is like a cookie cutter. The container is the actual cookie. You can stamp out many identical cookies (containers) from one cutter (image).

Container

A Container is a running instance of an image. It's isolated, has its own filesystem (based on the image layers plus a writable layer), its own network interface, and its own process space. Containers are ephemeral by default — when they stop, their writable layer is gone unless you use volumes.

Dockerfile

A Dockerfile is a text script of instructions telling Docker how to build an image. Each instruction creates a new layer. You'll master this in Module 04.

Registry

A Registry is a storage system for Docker images. Docker Hub is the public default. You can also run private registries (AWS ECR, GitHub Packages, self-hosted). Images are stored as name:tag, e.g. nginx:1.25.

Volume

A Volume is Docker-managed persistent storage. Containers are ephemeral — volumes let data outlive the container. Essential for databases, uploads, logs.

Network

Docker has its own networking subsystem. Containers on the same Docker network can talk to each other by name. Containers on different networks cannot — great for security segmentation.

The Lifecycle at a Glance

lifecycle
# 1. Write a Dockerfile
# 2. Build it into an Image
docker build -t myapp:1.0 .

# 3. Run it as a Container
docker run myapp:1.0

# 4. (Optional) Push Image to Registry
docker push myusername/myapp:1.0

# 5. On another machine — pull and run
docker pull myusername/myapp:1.0
docker run myusername/myapp:1.0
Quick Check: What is the relationship between an Image and a Container?
An image is a static, read-only template. A container is the live, running instance created from that image. Multiple containers can run from the same image simultaneously.

Docker Desktop is the easiest way to get started on Mac and Windows. Linux installs Docker Engine directly.

Mac / Windows — Docker Desktop

1
Download Docker Desktop
Go to docker.com/products/docker-desktop and download the installer for your OS.
2
Install and Start
Run the installer. Docker Desktop includes Docker Engine, CLI, Docker Compose, and a GUI dashboard.
3
Verify Installation
Open your terminal and run the commands below.

Linux (Ubuntu/Debian)

bash
# Remove old versions
sudo apt-get remove docker docker-engine docker.io containerd runc

# Install dependencies
sudo apt-get update
sudo apt-get install ca-certificates curl gnupg

# Add Docker's official GPG key
sudo install -m 0755 -d /etc/apt/keyrings
curl -fsSL https://download.docker.com/linux/ubuntu/gpg | \
  sudo gpg --dearmor -o /etc/apt/keyrings/docker.gpg

# Set up the repository
echo \
  "deb [arch=$(dpkg --print-architecture) signed-by=/etc/apt/keyrings/docker.gpg] \
  https://download.docker.com/linux/ubuntu \
  $(. /etc/os-release && echo "$VERSION_CODENAME") stable" | \
  sudo tee /etc/apt/sources.list.d/docker.list > /dev/null

# Install Docker Engine
sudo apt-get update
sudo apt-get install docker-ce docker-ce-cli containerd.io docker-compose-plugin

# Add your user to the docker group (no more sudo)
sudo usermod -aG docker $USER
newgrp docker

Verify Everything Works

bash
# Check Docker version
docker --version
# Docker version 26.x.x, build xxxxxxx

# Check Docker Compose version
docker compose version
# Docker Compose version v2.x.x

# Run your first container!
docker run hello-world
# Should print "Hello from Docker!" if all is working
⚠️
Linux users: After adding yourself to the docker group, log out and back in (or run newgrp docker) for it to take effect. Otherwise every docker command will require sudo.

# Tag an image (rename / add alias)

Working with Containers

bash — container commands
# Run a container (pulls image if not local)
docker run nginx

# Run in the background (detached)
docker run -d nginx

# Run with a name
docker run -d --name webserver nginx

# Run interactively (get a shell inside the container)
docker run -it ubuntu bash

# Map a port: host:container
docker run -d -p 8080:80 --name webserver nginx
# Now visit http://localhost:8080

# List running containers
docker ps

# List ALL containers (including stopped)
docker ps -a

# Stop a container
docker stop webserver

# Start a stopped container
docker start webserver

# Restart a container
docker restart webserver

# Remove a container (must be stopped)
docker rm webserver

# Force remove a running container
docker rm -f webserver

# Auto-remove container when it exits
docker run --rm ubuntu echo "Hello"

Inspecting Running Containers

bash — debugging
# View container logs
docker logs webserver
docker logs -f webserver    # follow (tail -f equivalent)

# Execute a command in a running container
docker exec -it webserver bash
docker exec webserver cat /etc/nginx/nginx.conf

# Copy files to/from a container
docker cp webserver:/etc/nginx/nginx.conf ./nginx.conf
docker cp ./myfile.html webserver:/usr/share/nginx/html/

# View resource usage
docker stats

# View container details
docker inspect webserver
💡
Pro tip: docker exec -it <name> bash is your best debugging tool. It drops you into a shell inside the running container so you can look around and diagnose issues — just like SSH-ing into a server.
Which flag runs a container in the background?
The -d flag stands for "detached" mode. The container runs in the background and returns you to your terminal prompt immediately.

Writing Dockerfiles

A Dockerfile is the recipe for your image. Every instruction creates a layer. Understanding layers is key to writing efficient Dockerfiles.

Dockerfile Instructions

InstructionPurposeExample

ADDLike COPY but can extract tars/fetch URLsADD app.tar.gz /app

ENVSet environment variableENV NODE_ENV=production
ARGBuild-time variable (not in final image)ARG VERSION=1.0

Real-World Example: Node.js App

Dockerfile
# Use a specific version tag — never use :latest in production
FROM node:20-alpine

# Set working directory inside the container
WORKDIR /app

# Copy package files FIRST (layer caching trick)
# If dependencies don't change, this layer is cached
COPY package*.json ./

# Install dependencies
RUN npm ci --only=production

# Copy the rest of the application code
COPY . .

# Set environment variable
ENV NODE_ENV=production

# Create a non-root user for security
RUN addgroup -S appgroup && adduser -S appuser -G appgroup
USER appuser

# Document the port the app listens on
EXPOSE 3000

# Health check
HEALTHCHECK --interval=30s --timeout=3s \
  CMD wget -qO- http://localhost:3000/health || exit 1

# Start the application
CMD ["node", "server.js"]

CMD vs ENTRYPOINT

This trips everyone up. Here's the definitive explanation:

💡
ENTRYPOINT — the fixed executable. Always runs. Not overridden by docker run arguments.

CMD — default arguments for ENTRYPOINT, or the default command if no ENTRYPOINT is set. Is overridden by docker run <image> <your-command>.

Together: ENTRYPOINT ["python"] + CMD ["app.py"] → runs python app.py. You can override CMD without touching ENTRYPOINT: docker run myimg script2.py → runs python script2.py.

Layer Caching — The Most Important Optimization

Docker caches each layer. If a layer hasn't changed, Docker reuses the cached version. Order your instructions from least frequently changing to most frequently changing.

❌ Bad Order (cache busted on every code change)
FROM node:20-alpine
COPY . .           # Copies ALL files including code
RUN npm install    # This runs every single time code changes!
✅ Good Order (dependencies cached separately)
FROM node:20-alpine
COPY package*.json ./   # Only package files change when deps change
RUN npm ci              # Cached unless package.json changed
COPY . .               # Code changes only invalidate THIS layer onward

Volumes & Persistent Storage

Containers are ephemeral — any data written inside a container is lost when it's removed. Volumes solve this.

Three Types of Mounts

💾
Named Volumes
Managed by Docker. Stored in Docker's area on the host filesystem. Best for production databases and persisting app data.
📁
Bind Mounts
Mount any host directory into the container. Perfect for development — code changes reflect immediately without rebuilding.
⚡
tmpfs Mounts
Stored in host memory only. Never written to disk. Use for sensitive temporary data like secrets or ephemeral caches.

Named Volumes

bash
# Create a volume
docker volume create mydata

# List volumes
docker volume ls

# Inspect a volume
docker volume inspect mydata

# Use a volume when running a container
docker run -d \
  -v mydata:/var/lib/postgresql/data \
  --name postgres \
  postgres:16

# Remove a volume
docker volume rm mydata

# Remove all unused volumes (dangerous!)
docker volume prune

Bind Mounts (Development Workflow)

bash
# Mount current directory into container
# Edit code locally → changes reflect inside container instantly
docker run -d \
  -v $(pwd):/app \
  -p 3000:3000 \
  --name dev-server \
  myapp:dev

# Read-only bind mount (container can't write to host)
docker run -d -v $(pwd)/config:/app/config:ro myapp
✅
Dev vs Prod pattern: Use bind mounts in development (instant code reload), named volumes in production (Docker manages the data lifecycle safely).

Volume in Docker Compose (Preview)

docker-compose.yml
services:
  db:
    image: postgres:16
    volumes:
      - pgdata:/var/lib/postgresql/data
    environment:
      POSTGRES_PASSWORD: secret

volumes:
  pgdata:   # Docker manages this

Docker Networking

Networking is how containers talk to each other and the outside world. Docker has a powerful built-in networking system.

Network Drivers

DriverUse CaseNotes
bridgeDefault for single-host. Containers on same bridge can communicate.Creates a virtual network on the host
hostContainer uses host's network directly (no isolation)Best performance, no port mapping needed
noneNo networking at allMaximum isolation
overlayMulti-host networking (Docker Swarm/K8s)Requires Swarm mode or plugin
macvlanContainer gets its own MAC address on physical networkAdvanced, legacy app support

Working with Networks

bash
# List networks
docker network ls

# Create a custom bridge network
docker network create mynetwork

# Run containers on the same network
docker run -d --name api --network mynetwork myapp:api
docker run -d --name db --network mynetwork postgres:16

# Now "api" can reach "db" just by name: postgres://db:5432
# Docker provides built-in DNS resolution within a network!

# Connect a running container to a network
docker network connect mynetwork mycontainer

# Disconnect
docker network disconnect mynetwork mycontainer

# Inspect a network
docker network inspect mynetwork

# Remove a network
docker network rm mynetwork

Port Publishing

bash
# -p HOST_PORT:CONTAINER_PORT
docker run -p 8080:80 nginx   # localhost:8080 → container:80

# Bind to specific host interface
docker run -p 127.0.0.1:8080:80 nginx  # Only accessible from localhost

# Publish all exposed ports to random host ports
docker run -P nginx

# Check port mapping
docker port mycontainer
💡
Key insight: Containers on the same user-defined network can reach each other by container name — Docker has built-in DNS. Containers on different networks are completely isolated from each other. This is how you implement network security segmentation in microservices.

Docker Compose

Docker Compose lets you define and run multi-container applications with a single YAML file. This is what you'll use every day.

A Complete Stack Example

docker-compose.yml — Full Stack App
version: "3.9"

services:

  # Frontend (React / Next.js)
  frontend:
    build:
      context: ./frontend
      dockerfile: Dockerfile
    ports:
      - "3000:3000"
    environment:
      - NEXT_PUBLIC_API_URL=http://localhost:8000
    depends_on:
      - api
    networks:
      - app-network

  # Backend API
  api:
    build:
      context: ./backend
    ports:
      - "8000:8000"
    environment:
      - DATABASE_URL=postgresql://user:password@db:5432/mydb
      - REDIS_URL=redis://cache:6379
    depends_on:
      db:
        condition: service_healthy
      cache:
        condition: service_started
    volumes:
      - ./backend:/app     # Bind mount for hot reload in dev
    networks:
      - app-network
    restart: unless-stopped

  # PostgreSQL Database
  db:
    image: postgres:16-alpine
    environment:
      POSTGRES_USER: user
      POSTGRES_PASSWORD: password
      POSTGRES_DB: mydb
    volumes:
      - pgdata:/var/lib/postgresql/data
      - ./db/init.sql:/docker-entrypoint-initdb.d/init.sql
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U user -d mydb"]
      interval: 10s
      timeout: 5s
      retries: 5
    networks:
      - app-network

  # Redis Cache
  cache:
    image: redis:7-alpine
    command: redis-server --save 60 1
    volumes:
      - redisdata:/data
    networks:
      - app-network

volumes:
  pgdata:
  redisdata:

networks:
  app-network:
    driver: bridge

Multi-Stage Builds

Multi-stage builds are one of the most powerful Dockerfile features. They let you use one stage to build your app and another to produce a tiny final image — without shipping build tools to production.

The Problem

If you install a compiler, build tools, dev dependencies just to compile your app, all that bloat ends up in your final image. A Go binary might be 10MB but the image with the compiler is 800MB. Multi-stage fixes this.

Go Application Example

Dockerfile — Go (800MB → 15MB)
# ===== STAGE 1: Build =====
FROM golang:1.22-alpine AS builder

WORKDIR /app
COPY go.mod go.sum ./
RUN go mod download

COPY . .
RUN CGO_ENABLED=0 GOOS=linux go build -a -installsuffix cgo -o main .

# ===== STAGE 2: Production =====
FROM scratch   # Literally empty image!

COPY --from=builder /app/main /main
COPY --from=builder /etc/ssl/certs/ca-certificates.crt /etc/ssl/certs/

EXPOSE 8080
ENTRYPOINT ["/main"]
# Final image: ~8MB instead of ~800MB!

React App Example

Dockerfile — React + Nginx
# Stage 1: Build the React app
FROM node:20-alpine AS builder

WORKDIR /app
COPY package*.json ./
RUN npm ci

COPY . .
RUN npm run build

# Stage 2: Serve with Nginx (no Node.js in final image!)
FROM nginx:1.25-alpine

# Copy built assets from previous stage
COPY --from=builder /app/dist /usr/share/nginx/html
COPY nginx.conf /etc/nginx/conf.d/default.conf

EXPOSE 80
CMD ["nginx", "-g", "daemon off;"]
✅
Rule of thumb for image sizes: Use alpine variants of official images (nginx:alpine, node:alpine, python:slim). Alpine Linux is a minimal distribution (~5MB) vs the default Debian-based (~100MB+).

CI/CD Integration

Docker is the backbone of modern CI/CD. Every pipeline builds images, tests them in containers, and ships them to a registry.

GitHub Actions Pipeline

.github/workflows/docker.yml
name: Build and Push Docker Image

on:
  push:
    branches: [main]
  pull_request:
    branches: [main]

env:
  REGISTRY: ghcr.io
  IMAGE_NAME: ${{ github.repository }}

jobs:
  build-and-push:
    runs-on: ubuntu-latest
    permissions:
      contents: read
      packages: write

    steps:
      - name: Checkout code
        uses: actions/checkout@v4

      - name: Set up Docker Buildx
        uses: docker/setup-buildx-action@v3

      - name: Login to GHCR
        uses: docker/login-action@v3
        with:
          registry: ${{ env.REGISTRY }}
          username: ${{ github.actor }}
          password: ${{ secrets.GITHUB_TOKEN }}

      - name: Extract metadata
        id: meta
        uses: docker/metadata-action@v5
        with:
          images: ${{ env.REGISTRY }}/${{ env.IMAGE_NAME }}
          tags: |
            type=sha
            type=ref,event=branch
            type=semver,pattern={{version}}

      - name: Build and push
        uses: docker/build-push-action@v5
        with:
          context: .
          push: ${{ github.event_name != 'pull_request' }}
          tags: ${{ steps.meta.outputs.tags }}
          labels: ${{ steps.meta.outputs.labels }}
          cache-from: type=gha
          cache-to: type=gha,mode=max

Security Best Practices

Security in Docker is a real concern. These practices are expected knowledge in any production environment.

👤
Non-Root User
Never run containers as root. Create a dedicated user in your Dockerfile.
🔖
Pin Image Tags
Use exact versions (node:20.11.1) or SHA digests. Never :latest in prod.
🔍
Scan Images
Use docker scout, Trivy, or Snyk to find vulnerabilities in your images.
🔒
Secrets Management
Never put secrets in Dockerfiles or images. Use Docker Secrets, environment files, or a vault.

1. Run as Non-Root

Dockerfile
FROM node:20-alpine

WORKDIR /app
COPY package*.json ./
RUN npm ci --only=production
COPY . .

# The node image already has a 'node' user — use it
USER node

EXPOSE 3000
CMD ["node", "server.js"]

2. Secrets — Do This, Not That

❌ NEVER DO THIS — secret baked into image history
ENV DATABASE_PASSWORD=supersecret  # Visible in docker history!
RUN curl -H "Authorization: mytoken" api.example.com  # Leaked!
✅ Pass secrets at runtime
# Pass as environment variable at runtime (from .env or CI secrets) docker run -e DATABASE_PASSWORD=$DB_PASS myapp # Using Docker secrets (Swarm) echo "supersecret" | docker secret create db_pass - docker service create --secret db_pass myapp # BuildKit secrets (for build-time only, not in image!) # In Dockerfile: RUN --mount=type=secret,id=npm_token \ NPM_TOKEN=$(cat /run/secrets/npm_token) npm install # Build with: DOCKER_BUILDKIT=1 docker build --secret id=npm_token,src=.npmrc .

3. Scan for Vulnerabilities

bash
# Docker Scout (built into Docker Desktop)
docker scout cves myapp:1.0
docker scout recommendations myapp:1.0

# Trivy (open source, excellent for CI)
trivy image myapp:1.0
trivy image --severity HIGH,CRITICAL myapp:1.0

# Snyk
snyk container test myapp:1.0

4. Read-Only Filesystem

bash
# Run container with read-only filesystem docker run --read-only \ --tmpfs /tmp \ # Allow writes to /tmp in memory --tmpfs /run \ myapp

5. Limit Resources

bash
# Limit CPU and memory docker run \ --memory="512m" \ --cpus="0.5" \ myapp # In docker-compose.yml services: api: deploy: resources: limits: cpus: "0.5" memory: 512M

Orchestration: Docker Swarm & Kubernetes

Single containers are great for dev. Production needs orchestration — automated deployment, scaling, healing, and load balancing of containers.

Docker Swarm (Built-in)

Docker Swarm is Docker's built-in orchestration. Simpler than Kubernetes, great for smaller teams.

bash — Docker Swarm
# Initialize a swarm (on manager node) docker swarm init # Add worker nodes (run on each worker) docker swarm join --token SWMTKN-1-xxxx manager-ip:2377 # Deploy a stack (like compose for swarm) docker stack deploy -c docker-compose.yml myapp # List services docker service ls # Scale a service docker service scale myapp_api=5 # View service logs docker service logs myapp_api # Rolling update docker service update --image myapp:2.0 myapp_api

Interview Q&A

These are real questions asked in Docker / DevOps interviews. Click each to reveal the full answer.

Easy What is Docker and why is it used? ▸
Docker is an open-source platform for developing, shipping, and running applications in containers. Containers package your application code with all its dependencies — libraries, runtime, environment config — into a single portable unit. This solves the "works on my machine" problem. Benefits include: consistent environments across dev/staging/prod, fast startup (seconds vs minutes for VMs), reduced overhead (shared OS kernel), and easy scaling. Docker is foundational to microservices, CI/CD pipelines, and cloud deployments.
Easy What is the difference between a Docker Image and a Docker Container? ▸
An image is a read-only, layered template — it's the blueprint. A container is a running (or stopped) instance of an image. Analogy: an image is a class definition; a container is an instance of that class. You can run multiple containers from the same image simultaneously. Images are stored in a registry and pulled to create containers. Containers have a writable layer on top of the image's read-only layers, which is lost when the container is removed (unless you use volumes).
Easy What is a Dockerfile? ▸
A Dockerfile is a text file containing a series of instructions that Docker reads to build an image. Each instruction creates a new read-only layer. Key instructions include: FROM (base image), RUN (execute commands), COPY/ADD (copy files), ENV (environment variables), EXPOSE (document port), CMD/ENTRYPOINT (startup command), and USER (set runtime user). You build a Dockerfile with docker build -t name ..
Easy What is Docker Compose? ▸
Docker Compose is a tool for defining and running multi-container applications using a YAML file (docker-compose.yml). Instead of running multiple docker run commands manually, you define all services, their networks, volumes, and configuration in one file and start everything with docker compose up. It handles service dependencies, networking (services can reach each other by name), and provides a single command to manage the entire stack. It's the standard for local development environments.
Medium Explain Docker layer caching and how to optimize it. ▸
Docker builds images layer by layer. Each Dockerfile instruction that changes the filesystem creates a new layer. If a layer hasn't changed since the last build, Docker reuses the cached version — dramatically speeding up builds.

The key rule: order instructions from least-frequently-changing to most-frequently-changing. For a Node.js app: copy package.json first, run npm install, then copy the source code. This way, dependency installation is only re-run when package.json changes, not on every code edit. Also, chain RUN commands with && to minimize layers and avoid cache-busting mid-layer.
Medium What is the difference between CMD and ENTRYPOINT? ▸
Both define what runs when a container starts, but they work differently:

CMD provides a default command that can be overridden by arguments passed to docker run.

ENTRYPOINT defines the fixed executable that always runs. Arguments to docker run are appended to it, not replace it.

Together: ENTRYPOINT ["python"] + CMD ["app.py"] — running docker run myimg executes python app.py; running docker run myimg other.py executes python other.py. ENTRYPOINT stays, CMD is replaced. Use ENTRYPOINT for the main command and CMD for its default arguments.
Medium What are Docker volumes and when do you use them? ▸
Volumes are Docker's mechanism for persistent storage. Containers are ephemeral — their writable layer is destroyed when removed. Volumes persist data outside the container lifecycle.

Three types: Named volumes (managed by Docker, best for production databases like Postgres, MySQL), Bind mounts (map a host directory into the container — ideal for dev workflow where you want code changes to reflect immediately), and tmpfs mounts (in memory only, for sensitive temporary data).

Use volumes whenever your container needs to read or write data that should survive the container's lifetime — databases, file uploads, log files, SSL certificates.
Medium How do containers communicate with each other? ▸
Containers on the same Docker network can communicate using container names as hostnames — Docker provides built-in DNS resolution. For example, if you have a container named db running PostgreSQL on port 5432, another container on the same network can connect to it at db:5432.

Containers on different networks cannot communicate unless explicitly connected. This provides security isolation — you can have a frontend network, a backend network, and a database network, with only the API having access to the database network.

In Docker Compose, all services are automatically placed on the same default network and can reach each other by service name.
Hard What is a multi-stage build and why is it important? ▸
Multi-stage builds use multiple FROM statements in one Dockerfile. Earlier stages are used for compilation/building and their tools never end up in the final image — you only copy the output artifacts into a clean, minimal final stage.

Why it matters: A Go app needs a ~800MB build environment (compiler, SDK) to compile but the resulting binary is ~10MB. Without multi-stage, your production image is 800MB. With multi-stage, it's ~15MB (binary + a minimal base). Smaller images mean: faster pulls, reduced attack surface, less storage cost, and faster deployments.

Example: FROM golang:1.22 AS builder → compile the app → FROM scratch → COPY --from=builder /app/binary /binary. The final image only has the binary.
Hard What are Docker security best practices? ▸
Key security practices: (1) Non-root users — never run containers as root; create a dedicated user with USER instruction. (2) Pin image versions — use exact tags like node:20.11.1 or SHA digests, never :latest in production. (3) Minimal base images — use alpine or distroless images to reduce attack surface. (4) Never bake secrets into images — pass them as runtime environment variables, use Docker Secrets, or use BuildKit's secret mounts for build-time secrets. (5) Read-only filesystems — run containers with --read-only where possible. (6) Resource limits — set memory and CPU limits. (7) Scan images — use tools like Trivy, Docker Scout, or Snyk in your CI pipeline. (8) Use .dockerignore to avoid leaking sensitive files into the image.
Hard What's the difference between Docker and Kubernetes? ▸
Docker (and Docker Compose) is a tool for building and running containers on a single host. It's excellent for development, single-server deployments, and small-scale production.

Kubernetes (K8s) is a container orchestration platform for managing containers across a cluster of machines. It handles: automatic scaling (add more replicas under load), self-healing (restart failed containers), rolling deployments (zero downtime updates), load balancing, service discovery, secrets/config management, and more.

The key relationship: Docker builds the images; Kubernetes deploys and manages those images at scale. In a K8s cluster, you define a Deployment that says "run 5 replicas of image myapp:1.0" and K8s figures out where and how to run them across your cluster. They're complementary, not competing.
Medium How would you reduce the size of a Docker image? ▸
Several techniques: (1) Use alpine or slim base images — node:20-alpine is ~50MB vs node:20 at ~350MB. (2) Multi-stage builds — build in a fat image, copy only the binary/output into a minimal final image. (3) Minimize layers — chain RUN commands with && and clean up in the same layer (apt-get install && apt-get clean && rm -rf /var/lib/apt/lists/*). (4) .dockerignore — exclude node_modules, .git, tests, docs. (5) Only install production dependencies — npm ci --only=production. (6) Remove dev tools and caches in the same RUN layer they're used. (7) Use distroless images for maximum minimalism.
Easy What is a Docker Registry? ▸
A Docker Registry is a storage and distribution system for Docker images. Docker Hub is the default public registry. When you run docker pull nginx, Docker fetches the image from Docker Hub. Images are organized as name:tag. Private registries (AWS ECR, GitHub Packages, Azure ACR, Google Artifact Registry) are used in production to keep proprietary images private and co-located with your cloud infrastructure. You authenticate with docker login [registry-url] and push/pull images as usual.
Hard How would you set up a CI/CD pipeline with Docker? ▸
A typical Docker CI/CD pipeline: (1) Trigger — pipeline fires on push to main or a PR. (2) Build — build the Docker image using docker build. Use layer caching (GitHub Actions: cache-from: type=gha) to speed up builds. (3) Test — run tests inside a container (docker run --rm myapp:test npm test) or use docker compose to spin up the full stack for integration tests. (4) Scan — run a vulnerability scan with Trivy or Docker Scout. Fail the pipeline on HIGH/CRITICAL vulnerabilities. (5) Push — tag the image with the git SHA and push to the registry (only on main, not PRs). (6) Deploy — trigger deployment (update a Kubernetes Deployment, ECS task definition, etc.) to pull the new image. (7) Verify — health check or smoke test the deployment.
